# 一、实训背景与目的

## 1 背景

随着大语言模型和智能软件开发技术的快速发展，围绕求职、职业规划、简历优化和面试训练的智能应用场景逐渐成熟。传统求职准备过程往往存在三个痛点：一是简历撰写质量高度依赖个人经验，很多学生不知道如何围绕目标岗位重构简历；二是岗位要求理解不充分，面对大量 JD 时难以快速提炼核心能力要求；三是面试准备过程碎片化，用户往往在多个平台之间来回切换，缺乏连续的分析、追问和训练闭环。

在这样的背景下，本项目以 “Career Coach 智能简历分析与职业辅导系统” 为题，尝试将 FastAPI、Vue 3、LangGraph、大语言模型、Redis Stack、阿里云 OSS 等技术整合起来，构建一个面向求职场景的综合性智能系统。系统不是单一的聊天工具，也不是单一的简历打分器，而是围绕“简历分析—岗位匹配—职业问答—测验训练”建立完整流程，体现现代智能软件开发中“业务流程 + AI 能力 + 工程实现”协同落地的特点。

本项目还具有较强的课程实践意义。它同时覆盖后端接口设计、前端页面开发、数据库建模、对象存储接入、向量检索、工作流编排、爬虫采集、文档整理与最终交付等多个环节，较好体现了《智能软件开发综合实训》强调的综合性、系统性和完整性要求。

## 2 目的

本项目的总体目的，是实现一套能够真正辅助求职准备过程的智能软件系统，并通过该项目完成课程实训的综合训练目标。具体来说，项目包含以下几个层面的目的。

第一，完成一个可运行、可展示、可继续扩展的完整系统。系统需要具备用户注册登录、简历上传分析、职业问答、测验生成、岗位知识库管理等核心能力，能够在答辩和展示过程中形成一条清晰的业务闭环。

第二，验证大语言模型在真实业务场景中的工程落地方式。项目并不满足于简单调用模型接口，而是通过 LangGraph 管道、Agent 工具调用、Redis 向量检索、结构化输出和 SSE 流式响应等方式，把模型能力变成可以被前后端稳定消费的工程组件。

第三，训练多模块协同开发能力。通过本项目的设计与实现，需要综合使用 Python 后端开发、Vue 前端开发、MySQL 数据建模、Redis 向量索引、OSS 文件存储、Swagger 接口展示、爬虫数据采集与 GitHub 成果归档等多方面能力，从而提升完整的软件工程实践水平。

第四，形成一份内容扎实、图表丰富、结构规范的实训报告。报告不仅需要说明“做了什么”，还需要清楚说明“为什么这样做、系统如何实现、有哪些成果、有哪些不足和改进方向”，从而满足课程交付要求。

# 二、实训内容

## 1 系统总体内容

本项目的核心内容是围绕求职场景构建一套 AI 职业辅导系统。系统以简历为核心对象，先通过文件上传、文本解析和 AI 分析生成结构化报告，再围绕这份报告继续进行问答与训练。相比传统静态管理系统，该项目强调的是从输入材料到结构化结果、再到后续行为延展的全过程。

当前系统主要包含五个业务模块：认证模块、简历模块、对话模块、JD 知识库模块和测验模块。它们共同构成从简历分析到职业教练问答、再到测验训练和评估的完整链路。

## 2 技术架构与开发环境

系统采用前后端分离架构。前端使用 Vue 3、Vite、Pinia 和 Vue Router 组织页面与状态；后端使用 FastAPI、SQLAlchemy Async 和 Pydantic 构建接口与数据校验；AI 层通过 LangChain、LangGraph、DeepSeek 和 Embedding 组件组织智能能力；数据层使用 MySQL 8 保存结构化业务数据，Redis Stack 保存岗位语义向量索引，阿里云 OSS 保存用户上传的 PDF 简历文件。

在开发环境方面，项目依赖 Python、Node.js、MySQL、Redis Stack 和 OSS 配置，前后端通过本地联调方式开发。由于 FastAPI 自带 OpenAPI 文档能力，后端接口在开发阶段可以直接通过 `/docs` 页面查看和测试，这也为最终报告提供了非常好的可视化展示入口。

## 3 核心功能模块

系统的核心功能模块可以概括如下。

认证模块负责用户注册与登录，并通过 `X-User-Id` 方案完成当前版本的资源归属校验。简历模块负责接收 PDF 文件、上传 OSS、提取文本并触发 AI 分析流程。对话模块负责基于简历分析结果进行职业问答，并通过 SSE 流式返回模型生成内容。JD 知识库模块负责录入和维护岗位数据，并将岗位要求文本写入 Redis Stack 向量索引。测验模块负责根据目标 JD、能力画像和优化建议生成面试训练题目，并在作答结束后生成评估报告。

这些模块之间并不是孤立存在的，而是围绕同一份简历分析结果逐步展开，因此整个系统具有较强的上下游关联性。

## 4 AI 工作流与工具链

项目的 AI 工作流主要由两类能力构成。第一类是结构化分析型能力，即对简历文本进行多阶段处理；第二类是工具增强型能力，即在职业问答中结合知识检索和互联网搜索补充模型能力边界。

简历分析部分通过 LangGraph 组织成 6 节点固定管道，节点分别负责：简历问题识别、能力画像提取、评分报告、维度分析、梯队建议和岗位 JD 检索。这一流程使模型输出不再是单一文本，而是成为前端可展示、对话可继承、测验可复用的结构化资产。

值得专门说明的是，项目中已经实现了 Tavily 兜底机制。当前在 `ai/workflow/pipeline.py` 中，若 Redis 向量库检索岗位 JD 数量不足，则会调用 Tavily 搜索进行补充；在 `ai/tools/search_tools.py` 中，Career Coach Agent 也可以通过 `web_search` 工具调用 Tavily，获取外部实时信息。这一点非常适合在报告中单独展示，因为它说明系统不只是本地向量检索，还具备“本地知识优先、外部搜索补充”的能力设计。

## 6 数据库与数据存储设计

系统采用“关系型数据库 + 对象存储 + 向量存储”三元方案。MySQL 保存用户、简历、会话、消息、JD 元数据和测验数据；OSS 保存 PDF 简历文件；Redis Stack 保存岗位文本的向量索引。这样的分层存储设计既符合各类数据的特征，也方便后续扩展和维护。

当前系统中最重要的结构化数据对象包括：`users`、`resumes`、`conversations`、`messages`、`jd_knowledge`、`quiz_sessions` 和 `quiz_questions` 七张核心表。其中 `resumes` 表承载了最多的 AI 结果字段，是后续问答和测验模块的重要上游数据源。`jd_knowledge` 则是 MySQL 与 Redis Stack 之间的桥接表，用于保存岗位要求文本与向量化状态。

# 三、实训步骤

## 1 需求分析与方案设计

项目实施的第一步是对业务需求进行梳理和收敛。最初需求较为宽泛，既希望系统能分析简历，也希望能做岗位匹配、职业问答和训练评估。为了保证在实训周期内能够完整落地，项目将范围收敛到“简历分析—问答—测验”三条主线，并据此形成系统初步设计文档、数据库草稿和模块拆分方案。

在这一阶段，还明确了技术路线：后端采用 FastAPI，前端采用 Vue 3，AI 编排采用 LangGraph，JD 检索采用 Redis Stack，文件存储采用 OSS，开发辅助资料通过设计文档、功能逻辑文档和 Swagger 页面统一整理。

## 2 后端基础框架搭建

第二步是后端工程骨架搭建。此阶段完成了 `main.py` 应用入口、数据库连接配置、统一响应结构、认证依赖、ORM 模型与 Pydantic Schema 设计，并将认证、简历、对话、知识库和测验五大模块拆分到独立路由文件中。通过这种结构化搭建，后续每个功能模块都能沿着相同规范继续扩展。

同时，FastAPI 自带的 Swagger 能力在这一阶段就已经体现价值，后续新增接口都能自动纳入文档页，便于联调和最终展示。

## 3 简历分析主链路实现

第三步是实现项目的核心链路，即简历上传和分析流程。后端首先完成文件校验和 OSS 上传，再通过 `pypdf` 提取 PDF 文本，随后调用 LangGraph 管道生成结构化分析结果。分析内容包括问题建议、能力画像、评分结果、维度报告、梯队建议和相关岗位检索结果，并统一回写到 `resumes` 表。

这一阶段奠定了系统的语义基础，因为后续问答和测验都依赖这份分析快照。

## 4 前端页面与交互实现

第四步是前端页面搭建与联动实现。项目先完成登录页和落地页，再重点实现简历上传与分析结果页，以及对话页与测验面板。前端通过 `ScoreGauge`、`SkillBars`、`DimensionReport`、`QuizPanel` 等组件，把后端的结构化数据转化为可读性较强的页面内容，并通过流程动画和交互细节提升用户体验。

此阶段还结合 OpenDesign 等设计工具对页面结构进行辅助规划，使最终页面不仅能展示结果，而且具有较清晰的视觉层次。

## 5 JD 知识库与 Redis 向量检索实现

第五步是围绕岗位知识构建检索能力。项目先通过爬虫工具采集岗位数据，并将结果整理到 `jd_data.json`；随后通过知识库服务把岗位文本写入 MySQL 和 Redis Stack。向量索引建立完成后，简历分析流程便可以根据能力画像和梯队建议检索相应岗位，实现更贴近真实 JD 的推荐。

这一步是系统从“只有模型常识”走向“结合外部岗位语料”的关键升级。

## 6 职业对话与 Tavily 兜底实现

第六步是实现职业教练对话功能。系统围绕简历分析结果自动创建对话会话，并在用户发送消息时把分析摘要和历史消息一并注入模型上下文。Career Coach Agent 在需要时可调用本地 JD 检索工具；当本地岗位知识不足或需要补充外部信息时，则通过 Tavily 进行 Web 搜索补充。

这一阶段使系统具备了明显的智能体特征，也让报告中可以明确说明“项目中已经实现 Tavily 作为找不到 JD 时的补充机制”。

## 7 测验模块实现

第七步是实现 AI 测验功能。系统根据目标 JD、能力画像和简历建议生成面试题，用户在前端逐题作答后，后端记录答案、判定正误并在结束时生成评估报告。通过这一模块，系统实现了从静态分析到训练反馈的闭环延伸。

## 8 联调测试与资料整理

最后一步是联调、测试与成果整理。前后端围绕 Swagger 文档进行接口验证，围绕页面流程进行人工联调，并将数据库设计、接口截图、爬虫数据、Redis 与 OSS 截图、GitHub 仓库截图、OpenDesign 原型图和 SVG 架构图等材料整理到最终实训报告中，形成完整交付。

从功能层面看，系统已经实现以下关键成果：一是用户注册登录能力；二是 PDF 简历上传、存储和结构化分析能力；三是围绕分析结果继续提问的职业教练问答能力；四是 JD 数据导入与语义检索能力；五是根据简历和岗位背景生成测验与测评报告能力。

这些成果最大的价值在于它们可以互相承接。例如，简历分析结果直接服务对话模块和测验模块；JD 知识库直接服务岗位推荐和工具查询；Swagger 页面又反过来成为接口层成果展示的重要窗口。

## 3 数据库成果与表结构分析

数据库设计是本项目的重要工程成果之一。当前系统围绕用户、简历、会话、消息、岗位知识和测验训练，共形成 7 张核心业务表，同时在 Redis Stack 中建立 JD 向量索引。下面分别对主要数据表结构进行说明。

### 3.1 users 用户表

`users` 表主要保存用户名、密码哈希、昵称、头像地址、状态和最后登录时间等信息，是所有用户级资源的归属基础。

### 3.2 resumes 简历表

`resumes` 表是项目最重要的数据表之一，除了保存简历文件名和文件地址外，还通过多个 JSON 字段保存 AI 分析结果，包括建议列表、能力画像、梯队建议、岗位检索结果、评分报告和维度报告等。

### 3.3 conversations 对话表

`conversations` 表记录职业教练对话会话本身，包括所属用户、关联简历、标题、消息数量和最后活跃时间等信息。

### 3.4 messages 消息表

`messages` 表记录会话中的具体消息，包括角色、内容和 token 消耗信息，为对话历史回放和成本统计预留了基础。

### 3.5 jd_knowledge 岗位知识表

`jd_knowledge` 表保存岗位来源元数据和要求正文，并通过 `vector_indexed` 字段标记是否成功写入 Redis 向量库。

### 3.6 quiz_sessions 测验会话表

`quiz_sessions` 表记录一次测验的总体信息，包括测验标题、题目数量、状态、总分和最终结构化报告。

### 3.7 quiz_questions 测验题目表

`quiz_questions` 表记录每一道测验题的题干、选项、答案、解释、标签、作答结果和顺序信息，是训练模块的重要支撑。

## 4 AI 与检索成果分析

AI 能力是本项目区别于传统管理系统的核心成果。项目不仅实现了 LangGraph 简历分析工作流，而且实现了 Agent 工具调用与 Tavily 兜底机制。当前系统能够做到：优先从本地 Redis 向量库检索岗位要求，当本地岗位覆盖不足时，再通过 Tavily 补充外部搜索结果。这样既提升了系统的可控性，也增强了面对不完整语料时的适应能力。

从工程角度看，这部分成果说明系统已经具备“结构化分析 + 外部信息增强 + 多阶段可追问”的智能软件特征。

## 5 爬虫、Swagger、GitHub 与工程资料成果

## 6 问题、不足与后续改进

虽然项目已经完成了较完整的主链路，但仍有一些需要进一步改进的地方。第一，当前认证方案仍采用 `X-User-Id`，后续应升级为 JWT 或更规范的认证机制。第二，当前 Redis 的主要职责集中在向量检索上，设计文档中规划的缓存、限流和异步任务能力尚未完全落地。第三，当前简历文件主要支持 PDF，后续可扩展到更多格式。第四，测验模块目前以单选题为主，未来可以继续扩展为多轮开放式面试流程。

这些不足并不影响当前实训项目的完成度，反而说明项目具备继续演进的空间。
