# 智能简历优化工作流（路线 B 完整版）

> 本文档描述一个基于 LangGraph 的简历优化工作流功能。
> 知识库内容选 **目标岗位 JD 库**（非简历示例），通过 RAG 实现能力→梯队→JD 差距的闭环分析。

---

## 一、功能定位与边界

### 1.1 功能概述

本功能是一个完整的简历优化工作流：用户上传简历后，依次完成能力评估、梯队建议、JD 差距分析，最终在节点⑤生成综合报告。报告中的 4 维度简历优化建议（弱动词替换 / 量化缺失 / 结构问题 / JD 差距）复用 `ai/prompts/resume.py` 中已定义的提示词，由工作流节点⑤直接调用，不再单列为独立模块。

### 1.2 不做的事

- 不做用户自定义对话（工作流是固定流程，非交互式）
- 不做简历示例检索（投入产出比低，4 维度分析已覆盖写法问题）
- 不做面试题生成（属于模块 02 面试模拟）

---

## 二、整体工作流

```
[简历文件]
     │
     ▼
① 解析简历 (parse_resume)        → resume_text
     │
     ▼
② 能力评估 (evaluate_ability)    → ability_profile
     │
     ▼
③ 梯队建议 (suggest_tier)        → tier_suggestion
     │
     ▼
④ JD 检索  (retrieve_jds)        → retrieved_jds (Top-K)
     │
     ▼
⑤ 综合报告 (generate_report)     → final_report
     │
     ▼
[入库 + 返回]
```

工作流为**线性无循环**结构（本场景不需要追问循环），用 LangGraph `StateGraph` 顺序连接 5 个节点。

## 三、LangGraph 状态与节点详细

### 3.1 工作流状态 State

```python
class WorkflowState(TypedDict):
    resume_text: str                  # 解析后的简历纯文本
    ability_profile: dict             # 能力画像（结构化）
    tier_suggestion: dict             # 梯队建议（结构化）
    retrieved_jds: list[dict]         # 检索到的 JD 列表
    final_report: dict                # 最终综合报告
```

### 3.2 节点 ①：解析简历 `parse_resume`

| 项 | 说明 |
|----|------|
| 输入 | 上传的 PDF 文件 |
| 输出 | `resume_text: str` |
| 实现 | pypdf 提取文本 |
| 备注 | 文件落 OSS，原文存 `resumes` 表（沿用现有表） |

### 3.3 节点 ②：能力评估 `evaluate_ability`

让 LLM 从简历抽取结构化能力画像。

**输出结构 `ability_profile`：**

```json
{
  "tech_direction": "后端",                    // 技术方向：后端/前端/算法/数据/全栈/测试/运维
  "experience_level": "L2",                    // L1初级 / L2中级 / L3资深 / L4专家
  "experience_years": 3,                       // 推断的工作年限
  "tech_stack": [                              // 技术栈及熟练度
    {"name": "Java", "proficiency": "熟练"},
    {"name": "MySQL", "proficiency": "了解"}
  ],
  "project_highlights": [                      // 项目亮点摘要（最多3条）
    "独立设计订单系统分库分表方案"
  ],
  "quantification_score": 6,                   // 量化成果水平 0-10
  "summary": "3年后端经验，主导过订单系统重构"  // 一句话能力总结
}
```

**实现要点：**
- 使用 `get_resume_llm()`（低温度，稳定输出）
- Pydantic 结构化输出（`with_structured_output`）
- 提示词放 `ai/prompts/ability.py`

### 3.4 节点 ③：梯队建议 `suggest_tier`

基于能力画像给出投递梯队与目标岗位。

> ⚠️ 不再使用「小厂/中厂/大厂」粗粒度三分，改为 **梯队 × 岗位类型** 组合，使下游检索有明确锚点。

**输出结构 `tier_suggestion`：**

```json
{
  "recommended_tier": "big_edge",              // startup / mid / big_edge / big_core
  "tier_label": "大厂边缘业务",                 // 中文标签
  "reasoning": "项目深度中等，量化指标偏少，建议从大厂边缘业务切入",
  "target_positions": [                        // 目标岗位列表（用于下游 JD 检索）
    {
      "position": "Java 后端开发工程师",
      "level": "P5",
      "tier": "big_edge"
    }
  ]
}
```

**梯队枚举：**

| 取值 | 中文标签 | 说明 |
|------|---------|------|
| `startup` | 创业公司 | 0-200 人，技术广度优先 |
| `mid` | 中型公司 | 200-2000 人，技术深度与广度并重 |
| `big_edge` | 大厂边缘业务 | 大厂非核心线/新业务，门槛相对友好 |
| `big_core` | 大厂核心业务 | 大厂核心岗位，要求高深度与高量化 |

**实现要点：**
- 输入为 `ability_profile`，无需重新读简历
- 提示词要求 LLM 给出 **2-3 个目标岗位**，覆盖主推荐 + 备选
- 目标岗位的 `tier` 字段作为下游 JD 检索的 metadata 过滤条件

### 3.5 节点 ④：JD 检索 `retrieve_jds`

获取目标岗位的 JD 用于下游差距分析。采用 **知识库优先 + 搜索工具兜底** 的两阶段策略，避免知识库语料不全时工作流失效。

**检索策略（两阶段）：**

**阶段 A — 知识库向量检索**
1. 构造查询文本：`tech_direction` + `target_positions[0].position` + 能力关键词
2. Embedding 后在 Redis Stack 向量库做 Top-K 检索（K=3~5）
3. 叠加 metadata 过滤：`tier ∈ target_positions[*].tier`
4. 若某梯队命中不足，回退放宽 tier 过滤

**阶段 B — 搜索工具兜底（仅当阶段 A 命中不足时触发）**
- 触发条件：阶段 A 检索到的 JD 数量 < 3，或目标岗位梯队完全无命中
- 调用自定义搜索工具（见 3.7），按 `target_positions` 拼接搜索关键词联网检索 JD
- 搜索结果结构化清洗后，与阶段 A 命中结果合并去重
- 合并后若仍不足，剩余岗位由节点 ⑤ 标注「未命中知识库，基于 LLM 内部知识分析」

**输出结构 `retrieved_jds`：**

```json
[
  {
    "jd_id": 12,
    "company": "字节跳动",
    "position": "Java 后端开发工程师",
    "tier": "big_edge",
    "tech_direction": "后端",
    "requirements": "熟悉 Spring/MySQL/Redis，有分布式经验...",
    "similarity": 0.82,
    "source": "knowledge_base"                    // knowledge_base / web_search
  }
]
```

**实现要点：**
- 用 `ai/vectorstore/` 封装知识库检索器
- 知识库检索器接口：`async def search_jds(query: str, tiers: list[str], k: int) -> list[dict]`
- 搜索工具接口：`async def search_jds_online(query: str, k: int) -> list[dict]`（见 3.7）
- 节点内编排：先查知识库 → 判断命中数 → 不足则调搜索工具 → 合并去重
- 每条结果标注 `source` 字段，供下游与报告区分来源

### 3.6 节点 ⑤：综合报告 `generate_report`

整合前序所有节点输出，生成最终可执行报告。

**输出结构 `final_report`：**

```json
{
  "ability_summary": "3年后端，主导订单系统重构，量化能力中等",
  "tier_advice": {
    "recommended_tier": "大厂边缘业务",
    "reasoning": "..."
  },
  "jd_gap_analysis": [                         // 与每个检索 JD 的差距
    {
      "jd_id": 12,
      "position": "Java 后端开发工程师 @ 字节跳动",
      "matched_skills": ["Java", "MySQL", "Spring"],
      "missing_skills": ["Redis", "Kafka", "分布式锁"],
      "experience_gap": "缺少高并发场景实战描述",
      "suggestion": "简历中补充 Redis 缓存与消息队列相关项目经验"
    }
  ],
  "resume_optimization": [                     // 4 维度优化建议（复用 resume.py 提示词）
    {
      "type": "missing_quantification",
      "location": "工作经历 - 美团 - 第2项",
      "issue": "缺少量化指标",
      "before": "优化了接口性能",
      "after": "优化订单查询接口，TP99 从 800ms 降至 200ms"
    }
  ],
  "action_plan": [                             // 行动建议
    "补充 Redis 缓存项目实战，可在简历中加入缓存击穿防护案例",
    "学习 Kafka 基础并在个人项目实践"
  ]
}
```

**实现要点：**
- 输入：`resume_text` + `ability_profile` + `tier_suggestion` + `retrieved_jds`
- `resume_optimization` 复用 `ai/prompts/resume.py` 的提示词，但 `target_jd` 改为检索到的 JD 拼接
- `jd_gap_analysis` 对每个 JD 单独对比，并在每条结果中保留 `source` 字段（知识库 / 联网搜索）
- 报告整体用结构化输出，避免格式混乱

### 3.7 搜索工具 `search_jds_online`（节点 ④ 兜底用）

自定义 LangChain Tool，当知识库检索命中不足时联网获取 JD。

**触发条件：** 由节点 ④ 内部判断，知识库命中数 < 3 或目标梯队完全无命中时调用。

**工具定义：**

| 项 | 说明 |
|----|------|
| 工具名 | `search_jds_online` |
| 输入 | `query: str`（岗位 + 技术方向关键词，如 "字节跳动 Java 后端 P5"），`k: int`（期望条数，默认 3） |
| 输出 | `list[dict]`，结构与 `retrieved_jds` 元素一致，`source` 字段恒为 `web_search` |
| 底层 | 接入第三方搜索 API（如 Tavily / SerpAPI / Bing），取招聘站点结果页 |
| 后处理 | LLM 清洗：从搜索结果摘要中抽取「公司 / 岗位 / 要求」三要素，结构化为 JD 对象 |

**输出示例：**

```json
[
  {
    "jd_id": null,
    "company": "腾讯",
    "position": "后台开发工程师",
    "tier": "big_edge",
    "tech_direction": "后端",
    "requirements": "熟悉 C++/Go，了解分布式系统，有高并发项目经验...",
    "similarity": null,
    "source": "web_search"
  }
]
```

**实现要点：**
- 工具代码放 `ai/tools/jd_search.py`
- 搜索 API Key 通过 `ai/config/settings.py` 配置（`SEARCH_API_KEY`、`SEARCH_PROVIDER`）
- 联网结果 **不入知识库**（避免污染语料质量），仅在本次工作流内使用
- 联网结果可选落库到 `jd_knowledge` 表，`vector_indexed=0`，供后续人工审核后向量化入库
- 工具需做超时控制（默认 8s）与异常兜底，失败时返回空列表，节点 ④ 据此降级

## 四、JD 知识库构建

### 4.1 语料来源

| 阶段 | 来源 | 数量 | 说明 |
|------|------|------|------|
| MVP | 手工整理 | 30-50 条 | 从 BOSS 直聘、拉勾等公开 JD 抽取，覆盖 4 梯队 × 主流方向 |
| 扩展 | 脚本批量导入 | 100+ | 后续支持 CSV/JSON 批量导入接口 |

### 4.2 JD 结构化字段

每条 JD 录入时需标注：

| 字段 | 类型 | 说明 |
|------|------|------|
| company | str | 公司名称 |
| position | str | 岗位名称 |
| tier | enum | startup / mid / big_edge / big_core |
| tech_direction | enum | 后端 / 前端 / 算法 / 数据 / 全栈 / 测试 / 运维 |
| requirements | str | 岗位要求正文（用于向量化与检索） |
| source_url | str | JD 来源链接（可选） |

### 4.3 向量化流程

```
JD 原文 → 按「职责段 + 要求段」切片 → Embedding → 存入 Redis Stack
                                          │
                                          └─ metadata: {jd_id, tier, tech_direction, company, position}
```

**技术选型：**
- Embedding 模型：BGE-M3（本地，中文友好）或 OpenAI text-embedding-3-small（按 .env 配置切换）
- 向量库：Redis Stack（RediSearch 模块），与设计文档一致
- 切片粒度：单条 JD 整体向量化（JD 通常 < 2KB，无需细切）

### 4.4 知识库管理接口（后续阶段）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/knowledge/jd` | POST | 录入单条 JD，触发异步向量化 |
| `/api/knowledge/jd/import` | POST | 批量导入 JD（JSON/CSV） |
| `/api/knowledge/jd/list` | GET | JD 列表 |
| `/api/knowledge/jd/{id}` | DELETE | 删除 JD（同时删向量） |
| `/api/knowledge/jd/{id}/status` | GET | 查询向量化状态 |

异步向量化用 Celery（设计文档已规划），MVP 阶段可先同步处理。

## 五、数据库新增表

### 5.1 `jd_knowledge`（JD 知识库元数据表）

| 字段名 | 类型 | 默认值 | 备注 | 索引 |
|--------|------|--------|------|------|
| id | BIGINT UNSIGNED | AUTO_INCREMENT | 主键 | PRIMARY |

| position | VARCHAR(100) | - | 岗位名称 | - |
| tier | VARCHAR(20) | - | 梯队枚举 | INDEX |
| tech_direction | VARCHAR(20) | - | 技术方向 | INDEX |

| created_at | DATETIME | CURRENT_TIMESTAMP | 创建时间 | - |

> 向量本体存 Redis，MySQL 只存元数据与原文。

### 5.2 `resume_workflows`（工作流执行记录表）

| 字段名 | 类型 | 默认值 | 备注 | 索引 |
|--------|------|--------|------|------|
| id | BIGINT UNSIGNED | AUTO_INCREMENT | 主键 | PRIMARY |
| user_id | BIGINT UNSIGNED | - | 所属用户 | INDEX |
| resume_id | BIGINT UNSIGNED | - | 关联 resumes.id | - |
| ability_profile | JSON | - | 能力画像 | - |
| tier_suggestion | JSON | - | 梯队建议 | - |
| retrieved_jd_ids | JSON | - | 检索到的 JD ID 列表 | - |
| final_report | JSON | - | 最终综合报告 | - |
| status | VARCHAR(20) | 'pending' | pending/running/done/failed | INDEX |
| created_at | DATETIME | CURRENT_TIMESTAMP | 创建时间 | - |

复合索引：`(user_id, created_at DESC)`，用于按用户查工作流历史。

## 六、接口设计

### 6.1 触发工作流

```
POST /api/resume/workflow
Content-Type: multipart/form-data
X-User-Id: 1
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| file | file | 是 | 简历 PDF，最大 10MB |

**成功响应：**

```json
{
  "code": 200,
  "message": "分析完成",
  "data": {
    "workflow_id": 1,
    "resume_id": 5,
    "final_report": { /* 见 3.6 final_report 结构 */ }
  }
}
```

> 工作流耗时可控（5 个 LLM 调用 + 1 次向量检索，约 10-20s），采用同步返回。后续如耗时增长可改为异步任务 + 轮询。

### 6.2 查询工作流历史详情

```
GET /api/resume/workflow/{workflow_id}
X-User-Id: 1
```

返回该工作流的完整 `final_report`。

### 6.3 工作流历史列表

```
GET /api/resume/workflow/list?page=1&page_size=10
X-User-Id: 1
```

返回当前用户的工作流执行历史（按时间倒序）。

## 九、关键风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| JD 语料不足导致知识库检索命中少 | 差距分析依据不足 | 节点 ④ 命中数 < 3 时自动触发搜索工具联网补全；搜索仍失败时节点 ⑤ 降级为纯 LLM 内部知识分析并标注来源 |
| 搜索工具调用失败或超时 | 兜底通路中断 | 工具内置 8s 超时 + 异常返回空列表，节点 ④ 据此降级；搜索 API Key 失效时告警 |
| 联网 JD 质量参差 | 差距分析偏差 | 联网结果经 LLM 清洗结构化；报告中标注 `source` 区分知识库/联网，便于人工复核 |
| Embedding 模型选型影响检索质量 | 知识库检索召回偏差 | MVP 用 BGE-M3，预留配置切换 |
| LLM 能力评估不稳定 | 梯队建议波动 | 低温度 + 结构化输出 + 提示词约束 |
